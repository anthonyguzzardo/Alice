# Gotchas

## Charter

This file documents landmines that genuinely belong in long-term memory. It is **not** a holding pen for bugs that should be fixed.

Entries belong here if they fall into one of four categories:

1. **Necessary friction** — constraints from a deliberate architectural choice (logical FKs, JSONB auto-parse, `COUNT(*)` returning bigint). The friction is the cost of a tool the project chose on purpose.
2. **Historical landmines** — bugs that have been fixed; documented to prevent regression. Must include the fix date and what the wrong behavior was, so future-you doesn't reintroduce it.
3. **Discipline rules** — filters or checks the code must always apply (`gated = false`, `invalidated_at IS NULL`). The rule itself is the gotcha.
4. **Philosophy-driven choices** — behavior tied to a stated principle in CLAUDE.md, with that principle cited explicitly. The behavior is intended even though it's surprising.

Entries do **not** belong here if they describe a bug as a feature. Phrases like *"semi-intentional"*, *"matches the pattern of X being best-effort"*, or *"intentional but counterintuitive"* are smells. If you find yourself reaching for that phrasing, stop: is the underlying behavior actually intended, or are you laundering a fix you should be making? Documentation that rationalizes a bug ossifies it — future-you reads the framing and trusts it. The signal pipeline early-return bug (line below) lived for weeks because GOTCHAS described it as "semi-intentional." Don't repeat that mistake.

When you fix a bug that previously had a rationalizing entry, rewrite the entry as a category 2 historical landmine — keep the institutional memory, drop the rationalization.

## Database

- **JSONB auto-parse trap.** postgres.js auto-parses JSONB columns into JS objects on read. But signal functions and callers expect JSON *strings*. Four functions in `libDb.ts` have manual re-stringify logic: `getSessionEvents`, `getLatestWitnessState`, `getDynamicalSignals`, `getMotorSignals`. If you add a new function that reads a JSONB column and the caller passes it to `JSON.parse()` or to a Rust function expecting a string, you will get `[object Object]` or a silent double-parse. Always check whether the caller expects a string or an object.

- **Calibration questions have `scheduled_for = NULL`.** Daily journal questions have a `scheduled_for` date. Calibration questions (source_id = 3) do not -- they're created on-demand without a scheduled date. If you JOIN calibration sessions on `scheduled_for`, you get zero rows. Match by `dttm_created_utc::date` instead. See `getSameDayCalibrationSummary()` for the pattern.

- **Question IDs 42, 63, 64 are intentional orphans.** These are early calibration sessions (2026-04-14, 2026-04-17) that have responses but no session summaries or session events. They predate the calibration event-logging pipeline. They are NOT bugs or transaction failures. The response text is real. The health endpoint explicitly excludes them from the "sessions missing summary" anomaly count.

- **Every migration file must begin with `SET search_path = alice, public;`.** Migrations are run via `psql -f` without a wrapper that sets the search path. If you omit it, the migration runs against the `public` schema and silently creates duplicate tables/columns in the wrong schema. Migrations 002-010 and 014-015 all include it. Migrations 011-013 were missing it (fixed 2026-04-23). The canonical schema file `dbAlice_Tables.sql` sets it at line 40.

- **No physical foreign keys.** The entire schema uses logical FKs only. JOINs work, but there are no cascade deletes and no referential integrity checks. Deleting a parent row silently orphans children. See CLAUDE.md for the cascade dependency map. When in doubt, don't delete.

- **Unquoted camelCase aliases get lowercased.** PostgreSQL lowercases unquoted identifiers. `SELECT col AS totalDurationMs` becomes `totaldurationms`. Always double-quote: `SELECT col AS "totalDurationMs"`. Every query in libDb.ts does this correctly; don't break the pattern.

- **`getSessionSummary` uses `sql.unsafe()`.** It's the only query that does this, because it interpolates the giant `SESSION_SUMMARY_COLS` constant string into the query. Every other function uses tagged templates. Don't copy this pattern for new queries without reason.

- **`COUNT(*)` returns bigint.** postgres.js returns `COUNT(*)` as a string (PG bigint). Every count query in libDb.ts casts with `COUNT(*)::int` to get a JS number. Forget the cast and you get string comparisons.

## Rust / Avatar Engine

- **HashMap iteration is nondeterministic in Rust.** Rust's `HashMap` uses randomized hash seeds, so iteration order changes between runs. If you iterate a `HashMap` to build a probability distribution and then sample from it with a seeded PRNG, the same seed produces different output across runs. This was a real bug in `avatar.rs` that broke reproducibility. The fix: build with `HashMap`, freeze into a sorted `Vec<(K, V)>`, sample from the sorted vec. See `MarkovChain` and `PpmTrie` for the pattern. If you add a new data structure that feeds into sampling, never iterate a `HashMap` on the sampling path.

- **`serde_json::from_str().unwrap_or_default()` silently eats parse errors.** If corpus or profile JSON is malformed, `unwrap_or_default()` returns an empty vec or default struct, and the function proceeds with garbage state. Avatar functions now use `SignalError::ParseError` to propagate parse failures. If you add a new function that deserializes JSON, use `.map_err(|e| SignalError::ParseError(...))` and `?`, never `unwrap_or_default()`.

- **Avatar `compute()` uses a time-based seed. Tests use `compute_seeded()`.** The public `compute()` generates variety by seeding from `SystemTime`. Tests must call `compute_seeded()` with a fixed seed to verify determinism. If you add a new generation function with randomness, provide a seeded variant for testing.

- **`default_profile()` is `#[cfg(test)]` only.** It was removed from production code when `unwrap_or_else(|_| default_profile())` was replaced with error propagation. Tests still use it. If you need a default profile in production, the caller should construct one explicitly.

- **`HoldFlight::from_stream` was misaligned prior to 2026-04-21.** Holds and flights were filtered independently, so rollover typing (next key pressed before previous released, producing negative flight times) caused `holds[k]` and `flights[k]` to refer to different keystroke events. This affected 100% of sessions and contaminated all stored transfer entropy values (130%+ average shift, 5 sign flips across 26 sessions). RQA was not affected (computed on IKI series, not hold-flight pairs). Fixed by filtering both together: only push when hold AND flight are valid for the same event. Snapshot preserved as `tb_dynamical_signals_pre_alignment_fix_20260421`.

- **Flight time upper bound `ft < 5000.0` is probably too aggressive for journal sessions.** A 5093ms flight in a deep reflection session is more likely a cognitive pause than noise. The threshold was inherited from IKI filtering where it's appropriate; flight times are structurally different (include evaluation pauses, read-back pauses, reformulation pauses). The threshold may need to be session-type-dependent (journal vs calibration). Follow-up analysis plan documented in INC-001 of `METHODS_PROVENANCE.md`. Do not bundle this change with the alignment fix.

- **Signal functions must be bit-reproducible on `aarch64-apple-darwin` with the pinned toolchain.** Introducing a HashMap on a computation path, a `.sum()` on floats without Neumaier, or a new call to `SystemTime` in a signal function will break CI. See `src-rs/REPRODUCIBILITY.md` for the full contract. The reproducibility check runs automatically on PRs touching `src-rs/**`.

- **Pre-2026-04-25, the canonical schema file `dbAlice_Tables.sql` was missing columns that existed in production.** Migrations 017-021 added 14 dynamical-signal columns (`effective_information`, `causal_emergence_index`, `optimal_causal_scale`, `pid_synergy`, `pid_redundancy`, `branching_ratio`, `avalanche_size_exponent`, `dmd_*x4`, `pause_mixture_*x3`) and 3 motor columns (`mse_series`, `complexity_index`, `ex_gaussian_fisher_trace`). The migrations applied cleanly to the live DB, but the project's documented rule — "rewrite the CREATE TABLE so the schema always reads as a complete, intact script" — was not followed. Detected and folded back during Phase 4.5 audit. **If you add a column via migration, immediately update the matching CREATE TABLE in `dbAlice_Tables.sql`** so a fresh `psql -f dbAlice_Tables.sql` produces the same schema as a fully-migrated DB. The new `tests/db/fieldMaps.test.ts` catches this class of drift between the schema file and field maps.

- **Pre-2026-04-25, the napi boundary used JSON strings for keystroke streams, profile, corpus, profile distance, and batch correlations.** Each call serialized in TS, parsed in Rust, and many also re-serialized return values. For a 5K-keystroke session, ~450KB of JSON was processed across 3 calls. Replaced by typed `#[napi(object)]` inputs with a `From`-impl boundary-mirror pattern (`KeystrokeEventInput`, `AvatarProfileInput`, `DigraphAggregateEntry`, `DigraphEntry`). The `digraph_latency_profile` storage format is intentionally preserved as `Record<digraph, latencyMs>` JSONB for backward compatibility with existing readers (`libProfile`, `libCrossSessionSignals`); the FFI is typed, but storage is collapsed to the legacy shape at the JSONB boundary. If you write a new analytical query against `digraph_latency_json`, expect Record format; if you write a new compute path, accept typed `Vec<DigraphEntry>` from the FFI.

- **Pre-2026-04-25, the TypeScript bindings hand-wrote the `NativeModule` interface.** Every Rust signature change required manually mirroring it in `libSignalsNative.ts`; forgetting was silent (TS compiled fine, runtime fields were undefined → null in DB). Replaced by auto-generation from `napi-derive` type-def output (`src-rs/index.d.ts`), with the TS layer importing `* as Native` and using `NullCoerced<T>` to express the `undefined → null` post-coercion. **If you see a hand-written interface mirroring `#[napi]` shapes reappear in TypeScript, that is a regression.** Edit the Rust annotations and rebuild; the d.ts regenerates.

- **`tb_dynamical_signals_contaminated_20260421` and `tb_dynamical_signals_pre_alignment_fix_20260421` are intentional audit artifacts, not orphans.** They preserve pre-fix values from the HoldFlight alignment bug for methods provenance (INC-001). Do not drop them. Do not "clean up" the schema by removing them.

- **Hetzner CCX silently mixes AMD EPYC Milan (Zen 3) and Genoa (Zen 4) hosts.** Hetzner does not let you pin the microarchitecture; CCX13/CCX23 instances land on either generation depending on capacity. Same Rust binary can take different vectorized FP paths across these uarchs (Genoa adds AVX-512), which would silently break bit-identity across the fleet. The mitigation: production Linux builds compile with `RUSTFLAGS="-C target-cpu=x86-64-v3"` (AVX2 + FMA + BMI2 baseline, present on both Milan and Genoa, no AVX-512). The CI `build-linux-x64` job sets this flag. If you change the production target without preserving this flag, the bit-identity claim no longer holds across the fleet — it becomes "bit-identical given the same EPYC generation," which is not the same guarantee.

- **Bit-identity claims must be backed by provenance rows.** A signal row without an `engine_provenance_id` (NULL = pre-2026-04-25 era) can be reproduced only by guessing which binary produced it from git history, which is not a guarantee. Rows written after 2026-04-25 by the worker pipeline always have provenance stamped — see `tb_engine_provenance` (binary SHA-256, CPU model, target_cpu_flag). Any analytical claim that "we re-ran the binary and got the same numbers" must cite the provenance row to be falsifiable. If you write a new signal save path that bypasses `libSignalWorker`, you must also call `stampEngineProvenance(questionId, getEngineProvenanceId())` after the saves complete.

## Observatory

- **Discovery badges: same-domain couplings capped at "provisional".** `libCouplingStability.ts` only analyzes emotion->behavior cross-domain pairs. Behavioral-only and semantic-only couplings pass through `pushCouplings()` in `synthesis.ts`, which hardcodes `strength: 'provisional'` because there is no stability data to promote them. If you see every same-domain coupling stuck at "provisional" regardless of sample size or correlation strength, this is why. Phase-two follow-up: extend the stability system to same-domain pairs. See INC-008 in `METHODS_PROVENANCE.md`.

- **Discovery critical-r gate suppresses most couplings at low n.** The discoveries section uses a dynamic significance threshold: `max(criticalR(n), 0.3)`. At n=10 (minimum for coupling computation), r_crit is 0.624. At n=25, it's 0.40. Most early couplings will be filtered out. This is intentional: a correlation of r=0.5 from 10 data points is not evidence of anything. If discoveries appear empty despite coupling data existing in the raw tables, check the entry count.

## Signal Pipeline

- **Semantic residuals are stored but NOT ghost-validated.** The reconstruction adversary validates motor and dynamical signals. Semantic residuals (ghost word salad vs real text) are a trivially explained difference, not a cognitive measurement. `behavioral_l2_norm` is the paper-reported aggregate (dynamical + motor + perplexity). `semantic_l2_norm` is stored for the Phase 2 self-referencing longitudinal baseline system (`libSemanticBaseline.ts`). `total_l2_norm` includes both families for backward compat but is NOT the paper-reported number. If you see semantic residuals that are large and consistent across all sessions, that is correct behavior, not a bug.

- **Pre-2026-04-25, `return` in signal family blocks silently skipped all downstream families.** In `libSignalPipeline.ts`, the dynamical, motor, and process blocks each had `if (!ds) return;` / `if (!ms) return;` / `if (!ps) return;` when the Rust engine returned null. The `return` exited `computeAndPersistDerivedSignals` entirely. The earlier rationalization that "if Rust is down, most families can't compute" was wrong: semantic signals (pure TS), semantic baselines, integrity, profile, and reconstruction all run on persisted state and don't depend on the failed family's output. The bug voided the documented "each family is computed independently" guarantee at three sites. Replaced with scoped `if (result) { ... }` blocks. If you see a bare `return` reappear inside `computeAndPersistDerivedSignals`, that is a regression — the function's contract requires that one stage's failure never short-circuit the rest.

- **Signal pipeline ordering matters.** Integrity must run BEFORE profile update (it compares against prior profile). Reconstruction must run AFTER profile update (it uses the current profile). Cross-session signals depend on motor signals being persisted first. Don't reorder blocks without understanding the dependency chain.

- **Rust engine failure is silent, not fatal.** If the `.node` native module isn't built, `hasNativeEngine` is false, all compute functions return null, and signal tables stay empty for that session. The session still saves. The health endpoint exposes `rustEngine: false`. Run `npm run build:rust` to fix. This is by design -- a missing measurement is better than a blocked session.

- **The `.node` file is platform-specific.** `alice-signals.darwin-arm64.node` only works on Apple Silicon macOS. It won't exist after a fresh clone on a different platform. The `createRequire` import in `libSignalsNative.ts` catches the load failure gracefully.

## Subject Auth + Encryption

- **Pre-2026-04-25, subject auth was a single static `invite_code` cookie.** `libSubject.getRequestSubject` read the `alice_subject` cookie, looked up `tb_subjects.invite_code`, and returned the subject. There were no sessions, no expiry, no rotation. Replaced by Path 2-lite (username + Argon2id password + opaque session token in `tb_subject_sessions`). The `invite_code` column on `tb_subjects` is now nullable and unused by new code; do not reintroduce reads against it. If you see new code calling `getSubjectByInviteCode` or setting an `alice_subject` cookie, that is a regression — use `loginSubject` + `verifySubjectSession` instead.

- **`ALICE_ENCRYPTION_KEY` is permanent for the deployment lifetime.** Subject responses are encrypted at rest via AES-256-GCM (`libCrypto.ts`). Lose the key and every encrypted row becomes permanently unreadable. There is no key rotation in v1, no recovery flow, and no fallback. The key lives in two places only: `/etc/alice/secrets.env` on Hetzner (root-owned, group-readable by `alice`) AND the operator's password manager. Both copies must survive. If you find yourself asking "where else can we put the key for redundancy" — the answer is the password manager, that's the redundancy. Never commit it, never log it, never include it in error messages.

- **`libCrypto` throws on tamper.** `decrypt()` throws on a modified ciphertext, modified nonce, wrong key, or stripped auth tag — there is no path that returns garbage. Callers MUST treat decryption as fallible (`try`/`catch`) and MUST NOT silently fall back to plaintext or to a default value. A tamper exception means the row was modified in storage, which is an integrity issue worth surfacing, not swallowing.

- **Session tokens are stored as SHA-256(token), not plaintext.** The cookie holds the raw 32-byte hex token; `tb_subject_sessions.token_hash` holds the hash. A DB leak does not grant active sessions because the attacker would also need the live cookie. If you write a new endpoint that needs a token, use `generateSessionToken()` which returns both `token` (for the cookie) and `tokenHash` (for the DB) — never persist the raw token directly.

- **Password reset deletes ALL sessions for that subject.** `resetPassword` invalidates every active token in `tb_subject_sessions`, forcing re-login on every device. This is intentional: a reset implies the credential changed, and any existing session predates the new credential. Do not "preserve the current session" as a UX optimization — it would let a stolen token survive a password rotation.

- **Owner endpoints are protected by Caddy HTTP Basic Auth, not by the app.** `/api/respond`, `/api/calibrate`, `/observatory`, `/` and similar owner-only paths are NOT gated by `src/middleware.ts`. The protection is in `deploy/Caddyfile` via `basic_auth @owner_paths` with `OWNER_BASICAUTH_HASH`. If you add a new owner-side endpoint, verify the path is excluded from the subject allowlist in the Caddyfile, or it will be wide open on `fweeo.com`. Future phase: session-based owner auth replaces the basic-auth layer.

## Submission Flow

- **Pre-2026-04-25, response and calibration submissions ran the post-save pipeline as a fire-and-forget async IIFE.** `/api/respond` and `/api/calibrate` both kicked off generation, witness rendering, embedding, and the derived-signal pipeline in a detached `(async () => { ... })()` after returning the HTTP response. If the Node process crashed (OOM, deploy, signal, restart) between the response and pipeline completion, every signal for that session was permanently lost. The session text survived in the DB; nothing re-ran the pipeline. Replaced by the durable `tb_signal_jobs` queue + `libSignalWorker.ts` poll loop. Job rows are inserted in the same transaction as the response/calibration save, so a job exists if and only if its session exists. Boot-time sweep re-queues anything left in `running` state. **If you see a fire-and-forget IIFE pattern reappear in an API route, that is a regression** — enqueue a job instead.

- **Pre-2026-04-25, calibration event storage ran outside the `saveCalibrationSession` transaction.** `saveSessionEvents` in `/api/calibrate` was called after `saveCalibrationSession` returned, so a failed event save left the calibration session row in place without its keystroke stream or event log. The earlier rationalization that "session data is sacred, derived data is best-effort" was wrong: keystroke streams are raw measurement input, not derived data — they are *the* signal source the Rust engine consumes. Fixed by extending `saveCalibrationSession` to accept an optional events row and persist it inside the same transaction. If you see a `saveSessionEvents` call appear outside a transaction in a calibration or response code path, that is a regression.

- **`updateDeletionEvents` is an UPDATE, not an INSERT.** It modifies the session summary row that `saveSessionSummary` just created. Must run AFTER saveSessionSummary within the same transaction. If you reorder the transaction block, this silently updates zero rows.

- **Pre-2026-04-27, subject journal submissions did NOT save raw event log or keystroke stream.** `subject/respond.ts` only wrote `tb_responses` + `tb_session_summaries`, while `subject/calibrate.ts`, `api/respond.ts`, and `api/calibrate.ts` all wrote `tb_session_events` too. The asymmetry was latent (zero subject journal entries existed at the time of the fix) but would have made every subject journal forever derived-only — no way to recompute signals when a formula changes. Fixed by mirroring the calibration path's event-save block inside the same transaction. **If you see a session-summary-only save with no companion `saveSessionEvents` call in any submission path, that is a regression** — the four submission paths must remain symmetric on raw-input capture.

- **Pre-2026-04-29, subject submissions did NOT enqueue signal pipeline jobs.** `subject/respond.ts` and `subject/calibrate.ts` carried docstrings claiming the contamination boundary forbade `enqueueSignalJob`, `computePriorDayDelta`, `computeAndPersistDerivedSignals`, and `snapshotCalibrationBaselinesAfterSubmit`. The boundary was over-stated: the actual constraint is "no synchronous LLM/Anthropic call," which is already satisfied by the absence of LLM calls in any submission pipeline (question generation lives in the operator-run corpus refresh). Daily delta and Rust signal compute are pure (DB+math, napi `.node`); they run on prod for owner already. The result of the over-stated boundary: every non-owner subject accumulated perpetual daily-delta + drift-snapshot pileup, and the operator was the only path to derive their signals via `npm run drain-subjects` running on a laptop. Fixed by INC-023: subject paths enqueue jobs symmetrically with owner. Embed continues to defer when TEI is offline (prod has no TEI; operator drains via `npm run embed`). **If you see a new submission path that skips `enqueueSignalJob` for any reason other than "this isn't a session at all," that is a regression** — the four submission paths must remain symmetric on pipeline enqueue.

- **Linguistic densities and MATTR are computed server-side, not from the client summary.** The `respond` and `calibrate` endpoints compute NRC densities, cognitive/hedging/first-person densities, MATTR, and sentence metrics from the response text on the server. The client sends raw `sessionSummary` fields for keystroke/timing data only. Don't look for these in the client payload.

## Embeddings

- **Embeddings require TEI running on port 8090.** `libEmbeddings.ts` calls `http://localhost:8090/embed` (configurable via `ALICE_TEI_URL`). The server is a CPU-only build of TEI at `~/.cargo/bin/text-embeddings-router`. Start with: `~/.cargo/bin/text-embeddings-router --model-id Qwen/Qwen3-Embedding-0.6B --dtype float32 --port 8090`. If TEI is not running, embedding calls fail gracefully and the response saves without an embedding.

- **Matryoshka truncation happens client-side.** TEI outputs 1024-dim vectors (model native). `libEmbeddings.ts` truncates to 512 dims and L2-renormalizes before storage. The schema is `vector(512)`. If you bypass `generateEmbedding()` and call TEI directly, you'll get 1024-dim vectors that won't match the schema.

- **Invalidated embeddings coexist with active ones.** `tb_embeddings` has an `invalidated_at` column. Rows from the previous embedding model are preserved but invalidated. All queries (HNSW search, topic matching, `isRecordEmbedded`) filter `invalidated_at IS NULL`. If you add a new query against `tb_embeddings`, include this filter.

- **Embedding failures are never fatal.** `embedResponse` runs inside the durable worker pipeline (`libSignalWorker.runResponsePipeline`), wrapped in try/catch. If TEI is down, the failure is logged and the rest of the pipeline continues — derived signals compute, semantic baseline z-scores are NULL for that session until the embed lands. Backfill later with `npm run backfill`. **Analytical implication:** `tb_embeddings` is not a reliable index of which responses exist. A response can be saved without a successful embedding. Any analysis that assumes "responses with embeddings = all responses" will be wrong. Phase 6d will move the embed step to its own retryable job kind so eventual consistency is automatic.

## Semantic Baselines

- **`tb_semantic_trajectory` z-scores are gated.** Rows with `gated = true` have z-scores computed from baselines with fewer than 10 sessions (MINIMUM_N). These z-scores are unreliable and stored only for completeness. Any analytical consumer (drift detection, CUSUM, trajectory visualization) MUST filter `WHERE gated = false`. Currently no consumer reads this table analytically; when one is added, enforce this filter.

- **Baselines exclude calibration sessions.** The semantic baseline backfill processes only non-calibration sessions (question_source_id != 3). Calibration text is prompted and does not represent organic thinking. Including it would contaminate the within-person baseline.

## Question Generation

- **Seed phase: first 30 days, generation is a no-op.** `runGeneration` checks `responseCount < SEED_DAYS` and returns early. During this phase, questions come from `libSeeds.ts` scheduled by the seed script. If you're testing generation and nothing happens, check response count.

- **`scheduleQuestion` uses `ON CONFLICT (scheduled_for) DO NOTHING`.** If a question already exists for tomorrow, generation silently does nothing. This prevents duplicates but also means re-running generation for the same day has no effect.

## General

- **`localDateStr()` was server-TZ-only pre-2026-04-28 and silently misrouted subject submissions to the wrong calendar day.** The function used `Date#getDate/getMonth/getFullYear` (server local TZ) and ignored `tb_subjects.iana_timezone`. The Hetzner box runs UTC; ash (`America/Chicago`, CDT = UTC-5) submitted her first journal at 21:49 Chicago time = 02:49 UTC the next day. The server's `localDateStr()` returned the next UTC day, so `today.ts` matched her to *tomorrow's* seed (question 126, scheduled for 2026-04-28) instead of today's (question 125, 2026-04-27). She wrote her day-1 entry, but the server filed it under day-2. The next day, when she logged in during Chicago daytime, the server still returned the same date string, refetched question 126, found `existing_response_text != null`, and showed the "Noted. Let it sit." done message — she perceived this as "no question today." Fix on 2026-04-28: extended `localDateStr(date, ianaTimezone?)` to use `Intl.DateTimeFormat({timeZone})` when the TZ is supplied, plus a TZ-independent `addDays(yyyymmdd, days)` helper for the seeder's calendar-day increments. Subject paths (`api/subject/today`, `api/subject/respond`, `seedUpcomingQuestions` via `create-subject`) now pass `subject.iana_timezone`. Owner paths keep server-local behavior because owner is `iana_timezone='UTC'`. Ash's schedule was repaired in-place: question 125 deleted, questions 126-154 shifted back one day so 126→2026-04-27 (her existing response stays linked to the prompt she actually saw) and 127→2026-04-28 (today's fresh question). The `iana_timezone` column has existed since the schema was introduced; the bug was that no caller read it.

- **`nowStr()` returned a bare (TZ-unqualified) timestamp pre-2026-04-28 and silently skewed `tb_responses.dttm_created_utc` by hours on a non-deterministic subset of journal rows.** The function returned `new Date().toISOString().replace('T', ' ').slice(0, 19)` — a string like `'2026-04-28 16:26:35'` with no `Z` or `+00:00`. When postgres.js sent that as a parameter for the TIMESTAMPTZ column, PG implicitly cast text → timestamptz **using the connection's session `TimeZone` setting**. Sessions inheriting `UTC` stored the value correctly; sessions whose upstream backend had `TimeZone = America/Chicago` (or any non-UTC value) interpreted the bare timestamp as local time and converted to UTC for storage, producing a `wallclock + offset` skew on disk. Because postgres.js holds a connection pool against Supabase's session pooler — and any prior client's `SET TIME ZONE` persists for the life of an upstream backend session — adjacent requests in the same Node process could land in different timezones. Surfaced 2026-04-28 when response 84 (owner journal) was stored as `21:26:35+00` while its same-transaction `tb_session_events` row was `16:26:35+00` (5h skew = CDT offset). `saveSessionEvents` and `saveCalibrationSession` rely on the column DEFAULT (`CURRENT_TIMESTAMP`), which always returns `+00` regardless of session TZ — that asymmetry is what made the bug visible. Fix on 2026-04-28: `nowStr()` now returns `new Date().toISOString()` directly (already `Z`-qualified) for the production path and `${_dateOverride}T12:00:00+00:00` for the simulation override, so PG never re-interprets via session TZ. Both `saveResponse` and `saveComment` (the two `nowStr()` callers) are fixed by the same one-liner. The simulation `_dateOverride` mechanism (production never sets it) is preserved — see comment block at `libDb.ts:58-63`. Historical rows written through the bug remain skewed; quantifying the affected subset is a `tb_responses.dttm_created_utc` vs `tb_session_events.dttm_created_utc` per `question_id` audit, not yet run.

- **Error log is a flat file, not a database table.** `data/errors.log` is appended by `utlErrorLog.ts` via `fs.appendFileSync`. For background work, errors land here AND on the failed job's `last_error` column in `tb_signal_jobs` (set by `markSignalJobFailed`). If `data/errors.log` doesn't exist, console-only errors vanish on restart, but durable job rows still preserve the last failure reason for queue-level diagnostics.

- **`npm run dev` builds Rust first.** The dev script runs `./src-rs/build.sh && astro dev`. If the Rust build fails (missing toolchain, compile error), the dev server doesn't start. If you're only working on frontend/API code and don't need signals, you can run `astro dev` directly, but signal computation will be disabled.

- **Archival means deletion, not stubbing.** When features are archived, their tables, API handlers, lib functions, and downstream integration points are all deleted in the same commit. Data goes to `zz_archive_*` tables in the database. Don't leave stub functions that return hardcoded values.
